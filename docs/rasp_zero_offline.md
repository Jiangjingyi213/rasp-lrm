# RASP-Zero Offline Evaluation

This stage evaluates the first reasoning-aware pruning policy without claiming runtime acceleration. It reuses the completed Qwen3 counterfactual tables from GSM8K and MATH500.

## Purpose

The evaluator answers:

> At a comparable average pruning budget, does a reasoning-aware policy cause fewer answer flips than static, entropy-only, confidence-only, and probe-only policies?

Each row in the compact `05_probe_dataset.jsonl` tables is derived from one counterfactual ablation result:

```text
(dataset, problem, reasoning segment, module, ratio, layer set)
```

The offline evaluator chooses one previously measured action for each reasoning segment and reports its observed answer-flip risk.

The compact probe tables are used instead of `03_counterfactuals.jsonl` because policy evaluation does not need the long generated continuation text. This reduces I/O and avoids transferring very large JSONL files unnecessarily.

This is a policy simulation. It is not a real speed benchmark. The current counterfactual masks are diagnostic hooks rather than exported compressed models.

## Two-Step Workflow

### 1. Generate problem-level OOF risk scores

`src.main_generate_oof_probe_scores` trains a hidden-state linear risk probe with problem-level folds. Every score is generated by a model that did not train on that problem.

This matters because all actions and segments from one problem are correlated. A row-level split would leak information across train and validation sets.

The score estimates segment criticality:

```text
risk_score = P(answer flip after pruning | current reasoning state)
```

The first version intentionally uses an action-agnostic score. Rows for the same segment are averaged before routing. The estimator answers whether a segment is generally fragile; the router then decides how aggressively to prune it.

Before training, `src.main_validate_rasp_inputs` verifies that every formal shard is complete. It checks the expected compact-table size recorded by:

```text
03_counterfactuals.oracles.json
```

It also verifies that every executed reasoning segment has the complete `5 modules x 3 ratios` action grid. If a transferred JSONL file was truncated, the pipeline stops with the exact file and line number instead of producing a partial result.

### 2. Evaluate budget-matched policies

`src.main_rasp_zero_offline` compares:

- `dense_lrm`
- `static_mlp_block`
- `entropy_only`
- `confidence_only`
- `hidden_probe`
- `rasp_zero_stage`
- `prompt_router_safe_oracle`
- `step_safe_oracle`

For dynamic policies, lower-risk segments receive pruning budget first. RASP-Zero additionally caps aggressive pruning by reasoning stage:

| Stage | Maximum ratio |
|---|---:|
| verification | 0.2 |
| final | 0.2 |
| derivation | 0.4 |
| planning | 0.6 |
| understanding | 0.6 |

The evaluator reports both target and realized average ratios. A conservative stage policy may realize less pruning than the requested target.

## Interpretation Rules

- Lower `selected_action_flip_rate` is safer.
- Compare policies at similar `average_pruning_ratio`, not only by policy name.
- `average_pruning_strength_proxy` is not measured latency or FLOPs.
- `step_safe_oracle` is a diagnostic upper bound for safe routing.
- The motivation oracle in `03_counterfactuals.oracles.json` is different: it deliberately maximizes answer flips to expose structural sensitivity.

## Current Scope And Next Step

The current action bank uses existing counterfactual actions at ratios `0.2`, `0.4`, and `0.6`. These are useful for method validation but are too aggressive for the first runtime implementation on Qwen3-1.7B.

If the offline result supports RASP-Zero, the runtime version should start with:

```text
MLP-focused masks
ratios 0.0 / 0.05 / 0.10 / 0.20
route updates every 16 or 32 generated tokens
dense fallback for fragile windows
```

Attention and layer skipping should remain analysis-only until the MLP runtime path is stable.
